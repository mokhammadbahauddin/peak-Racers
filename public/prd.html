Section 01

Project Overview & Vision

What this product is, why it exists, and what it must accomplish.

🏎️
Problem Statement
The core pain we're solving

Most arcade racers prioritize intensity over aesthetics, creating a gap for players seeking "Cozy-Arcade" gameplay — high-energy kart mechanics wrapped in a soothing, premium pastel visual shell. Pastel Peak Racers fills this niche with a digitally tactile experience where every drift, boost, and victory feels physically satisfying.

🎯
North Star Vision
What "done" looks like

Deliver the definitive digitally-tactile racing experience. A player picking up the game for the first time should feel the premium weight of the cars, the satisfying "juice" of power-ups, and the dreamlike peace of the pastel world — all within 90 seconds of loading.

🎮
Target Platform

Web browsers (desktop primary, mobile responsive). Progressive enhancement for touch controls. No native app in v2.0.

💰
Business Model

Free-to-play core loop. Monetization via cosmetic skin packs (Rare / Epic / Legendary tiers) and Battle Pass. No pay-to-win mechanics.

🏆
Competitive Positioning

Occupies the whitespace between Mario Kart (competitive/Nintendo) and Monument Valley (serene/puzzle). Neither exists in browser-native form.

Section 04

Design Decisions Before Coding

Ordered by cost-to-change. Layer 1 decisions constrain everything below them. An ADR was written for each.

⚠ Rule: System design and data model must be stable before UX design begins. Skipping this order caused two major redesigns in v1. It does not happen in v2.
1
Most Expensive to Change
Data Model & Schema
Finalized before any API or UX work. Core entities: Player, Vehicle, Track, Race, Leaderboard, Upgrade, Skin. All cardinalities resolved (see Section 05). Indexes designed for leaderboard query patterns. Soft-delete + audit timestamps on all entities.
System Architecture
Decision: Monolith. Three.js core handles synchronous gameplay. Async background workers handle leaderboard recalculations and reward distribution. No microservices in v2 — the overhead is unjustified at this scale. Revisit at 1M MAU.
2
Infrastructure & Stack
Cloud: AWS (us-east-1 primary, eu-west-1 replica for GDPR). Runtime: containerized Node.js on ECS Fargate. CI/CD: GitHub Actions on day 1. CDN: CloudFront for all static game assets. No manual deploys — ever.
ADR-002 Written
API Contracts
OpenAPI 3.1 spec authored before frontend development. Frontend mocks against spec immediately. Versioned via URL prefix /v2/. All error responses follow RFC 7807 (Problem Details). See Section 11.
3
Security Model
Auth: JWT (15min access token + 30d refresh). AuthZ: RBAC (Player / Admin). Secrets: AWS Secrets Manager — zero hardcoded keys. Encryption: TLS 1.3 in transit, AES-256 at rest for PII. GDPR: right-to-erasure endpoint implemented before launch.
UX Flows & States
Every screen accounts for 5 states: Happy Path, Loading/Skeleton, Empty State (no cars owned), Error (connection lost), and Edge Case (10,000 leaderboard entries, network timeout mid-race). No screen may ship without all 5 states designed.

Section 06

UX States & Error Handling

Every screen must handle all five states before it ships. "We forgot to handle errors" is not a valid postmortem finding.

✅
Happy Path
Data loaded, user logged in, race available. Primary flow. Designed first.
⏳
Loading
Skeleton screens and loading car animation. Max perceived wait: 800ms before progress bar appears.
📭
Empty State
No vehicles owned, no races completed, new account. Must guide user to first action, not show blank UI.
⚡
Error State
Connection lost mid-race, API failure, payment error. Always include retry action + human-readable message.
🔀
Edge Cases
10,000 leaderboard entries, 0-coin balance at shop, network timeout at lap 3. All must be handled gracefully.

Observability & Structured Logging

Log Format (JSON)

Every log line includes: correlation_id, player_id (if authenticated), event_type, severity, timestamp_utc, service. No free-form strings. Parsed by CloudWatch Insights.

Alert Thresholds

P95 API latency > 300ms → PagerDuty warning. Error rate > 1% over 5min → PagerDuty critical. Leaderboard sync lag > 120s → Slack alert. CDN cache hit rate < 90% → engineering ticket.

Section 12

Risks & Mitigations

Write the mitigation next to each risk. A PRD with no risks is a PRD where someone didn't ask hard enough questions.

Severity Risk Description Mitigation Owner
HIGH Three.js performance on low-end hardware Target 60FPS may not be achievable on older Intel integrated graphics. Could break core experience for 30% of users. LOD system with 3 quality tiers (auto-detected). Manual quality selector in Settings. Tested on 2017 hardware in CI. Engine Lead
HIGH Leaderboard race submission cheating Client-submitted race results could be manipulated. Compromises competitive integrity and ruins the game for The Completionist persona. Server-side physics replay validation for top-100 submissions. Statistical anomaly detection (lap times 3σ below track average). Soft-ban on anomalies, manual review queue. Backend Lead
HIGH GDPR compliance at launch Missing data residency or erasure endpoint at launch creates legal liability in EU market. EU is 35% of addressable market. Legal review 4 weeks before launch. Erasure endpoint (FR-07) is a hard dependency for go-live. DPA signed with all sub-processors before any EU data flows. Product Lead
MED Third-party payment processor downtime Shop purchases fail during payment processor outage. Revenue impact + user frustration. Graceful degradation: shop shows "Purchases temporarily unavailable" without error codes. Queue failed transactions for retry within 24h. No double-charge risk. Backend Lead
MED Art style regression during dev Without a design token system, individual developers may deviate from the glassmorphism language, creating visual inconsistency. Tailwind config locked at design review. PR checklist includes screenshot comparison against Figma spec. Design token library treated as dependency, not a guideline. Design Lead
LOW Scope creep from stakeholder requests New features requested post-PRD sign-off could delay launch. Common in games where "just one more track" is easy to rationalize. Formal scope change request process with Product Lead sign-off and explicit sprint impact statement. Out-of-scope list (Section 07) is contractual. Product Lead
LOW Mobile performance on low-RAM devices Three.js on mobile browsers may run poorly on devices with <3GB RAM, affecting the commuter persona. Mobile-specific asset bundles (50% polygon reduction). Auto-detect mobile and apply low-quality preset. Explicit "Desktop recommended" notice for mobile users at first load. Engine Lead

Appendix B

Architecture Decision Record — ADR-001

Template used for all significant architectural decisions. ADR-001 is the monolith vs microservices decision.

Title
Use monolithic architecture for v2.0 (defer service extraction)
Status
ACCEPTED
Context
Team is 6 engineers. Expected scale is <5K concurrent at launch. Microservices introduce network overhead, distributed tracing complexity, and deployment complexity that our team cannot absorb without slowing velocity.
Decision
Build as a modular monolith. Game logic, auth, leaderboard, and shop are separate modules within one deployable. Async workers (leaderboard recalc, reward distribution) run as separate processes but share the same DB.
Alternatives
A) Full microservices — rejected: deployment complexity exceeds team capacity. B) Serverless functions — rejected: Three.js game server requires persistent WebSocket connections (v3 multiplayer). C) Monolith with separate leaderboard service — deferred to 1M MAU trigger.
Consequences
Faster development velocity. Single deploy artifact. Easier debugging. Tradeoff: single scaling unit — at 1M MAU the leaderboard module must be extracted. This is the documented trigger for service extraction.

Developer Truth / Brutal Honesty Log

A candid game developer assessment of the current state of Pastel Peak Racers v2.0.

Architecture Pillar The Brutal Truth Next Steps / Tech Debt to Fix
Track & Level Design The track is a "fake" Catmull-Rom spline (a generated tube). It lacks banked curves, jumps, or real organic level design. Collisions are just invisible walls pushing the player towards the spline's closest `t` parameter. It feels on-rails. FIXED Rebuilt the Cannon.JS physics Trimesh collider to dynamically include the extruded track walls and curb geometries. Removed artificial minDist pushback. Cars now physically bounce off real rigid track boundaries.
AI Opponents AI cars are literally on-rails. They sweep along the `trackCurvePts` by manually incrementing their `t` parameter. They have no physical bodies in the Cannon.js world, don't steer, don't collide with items or players, and don't experience physics. FIXED Verified AI currently uses full ArcadeVehicle Cannon.js simulation wrapper with steering logic rather than simple teleporting.
Item System The item system is visually okay but mechanically hollow. Shooting a shell or dropping a banana doesn't spawn a hazard—it just plays a particle effect and gives a mini speed boost. Items do not interact with other racers. FIXED Implemented Projectile sub-system in ItemManager. Green shells travel physically and bananas rest on the track. Both trigger spinout behaviors on AIs and the real player block.
Game Feel & Physics The hop and drift system is surprisingly good! Tire scrub sparks, conditional mini-turbos, and dynamic camera shake add strong "juice". However, the base physics hover system is a bit stiff, and drifting doesn't quite rotate the chassis mesh deeply enough. FIXED Implemented conditional chassis roll/pitch quaternion mixing in ArcadeVehicle.`render()` based on local lateral velocity.
Controls (Mobile vs. PC) Mobile buttons linearly map to `ArrowLeft`/`ArrowRight`. This makes tapping feel extremely jerky. True joysticks or smoothed turn interpolation is missing. FIXED Added a custom Touch-Analog Joystick overlay (`AnalogJoystick`) allowing fractional steering values and smooth interpolation.
Mario Kart-Tier Rework The game felt a bit generic and lacked the fast-paced, highly responsive feel of top-tier arcade racers. The visuals, while good, were missing modern post-processing and material depth. The AI was too passive. FIXED Completely reworked the physics for tighter, "Mario Kart"-like drifting. Upgraded RenderPipeline with ACESFilmicToneMapping and SSAO. Implemented true homing logic for rocket items and aggressive rubber-band AI.
